A

활성 주문번호 조회 타이밍

AI flag 수신 후 즉시 실시간 조회할지, 배치(N분 후)로 처리할지 결정 필요.
배치 방식이면 detected_at과 실제 조회 시점 사이에 lag이 생기고, 이후 24h 취소·환불 조건 판단 타이밍도 같이 밀릴 수 있음.
실시간 조회 기본 + API 타임아웃 등 예외 케이스 처리 방식을 별도 정의하는 방향 권장.

B

복수 주문 처리 기준

동일 channel_id에 활성 주문이 N개 존재할 때 어느 주문에 페널티를 부여할지 기준 미결.
후보: ① 최신 주문 1건 기준 (단순) ② 전체 주문에 각각 부여 (중복 과다 가능) ③ 운영 판단 대기 큐 이관 (수동 코스트 발생)
① 최신 주문 기준 + 나머지 DB 기록 보존이 현실적.

C

1

스코어 산출 기준 정의

정의 필요 페널티 점수(-3점 등)와 별개로 AI 탐지 횟수 기반 리스크 스코어를 판매자 단위로 누적.
예시 가중치: 탐지 1회 = +1점 / 24h 내 중복 탐지 = +0.5점 / 최근 30일 롤링 집계
일정 기간(예: 90일) 탐지 없으면 자동 감소하는 decay 로직도 함께 설계 권장.

2

노출 제한 단계 설계

임계값 결정 필요 리스크 스코어 구간별로 검색·추천 노출을 단계적 감소.
예시: 스코어 3~5 → 노출 30% 감소 / 6~9 → 노출 60% 감소 / 10이상 → 노출 완전 배제 + 어드민 알림
제재(-12점) 미달 회색지대 어뷰저를 커버하는 것이 핵심 목적.

3

검색/추천 알고리즘팀 인터페이스

협업 필요 Trust/Policy 측에서 판매자 uid별 리스크 스코어를 API 또는 DB 테이블로 제공, 검색/추천팀이 랭킹 로직에 반영하는 구조.
Trust는 스코어 산출·관리만 담당, 노출 로직은 알고리즘팀 소유로 역할 분리 가능.

4

판매자 이의제기 처리

정책 설계 필요 노출 제한을 사용자에게 비노출(비고지)로 운영할지, 어드민 내 이력 열람 가능 형태로 운영할지 결정 필요.
AI 오탐 케이스 대비해 reason 텍스트를 어드민에 저장하는 구조와 연계. 기존 페널티 이의제기 프로세스와 동일 채널 통합 운영 권장.
